= Towards Automated Circuit Discovery for Mechanistic Interpretability
# 문제
기존의 [[LLM]] 해석 연구는 사람이 직접 실험 설계해서 진행하고, 결과 보고 수동으로 하나하나 다 하는데 너무 느리고 복잡하다.
# method
계산 그래프에서 노드가 아니라 엣지를 중심으로 본다.
엣지들을 하나씩 지워가면서 이 작업에 뭐가 중요한지 찾아내는 것. 

## 알고리즘
#### **Step 1: 초기화**

- 모델의 모든 연결(Edge)이 포함된 전체 그래프에서 시작합니다.
    
- 사용자가 허용할 오차 범위인 **임계값(Threshold, $\tau$)**을 설정합니다.
    

#### **Step 2: 레이어별 순회 (Backward Pass)**

- 가장 마지막 층부터 거꾸로 한 단계씩 올라가며 각 엣지를 검사합니다.
    
- 현재 검사 중인 엣지를 '패칭(오염된 값으로 대체)' 해봅니다.
    

#### **Step 3: [[Path Patching]]
1. 두 가지 입력 준비: 
	- 깨끗한 입력(Clean Input): 모델이 정답을 맞히는 정상적인 데이터 (예: "The capital of France is Paris").
    
    - **오염된 입력(Corrupted Input):** 모델이 혼란을 느끼게 하는 데이터 (예: "The capital of Germany is Berlin").
        
2. **엣지 차단(Intervention):** 특정 노드 A에서 노드 B로 가는 정보(엣지)를 '오염된 입력'을 넣었을 때의 값으로 강제로 교체(Patching).
    
3. **영향력 측정:** 이렇게 정보를 바꿨을 때, 최종 출력값(Logits)이 정답에서 얼마나 멀어지는지 확인합니다.
 
4. **손실 함수(Loss Function) 계산:** [[KL-Divergence]] 등을 사용해 전체 모델의 출력과 엣지가 제거된 상태의 출력 차이를 계산합니다.

5. **판정:** * $\text{차이} < \tau$: 성능에 큰 영향이 없으므로 해당 엣지를 **영구적으로 제거**합니다.
    
    - $\text{차이} \geq \tau$: 해당 엣지는 회로의 핵심 구성 요소이므로 **보존**합니다.
        

#### **Step 4: 수렴할 때까지 반복**

- 더 이상 제거할 수 있는 엣지가 없을 때까지 이 과정을 반복합니다. 결과적으로 해당 태스크를 수행하는 데 꼭 필요한 '최소한의 연결망'만 남게 됩니다.

# 평가
## 성과
1. 연구자 직관 개입 없이 객관적인 결과 추출
2. 효율성
### 논문에 있는 발견 예시
- **간접 목적어 식별(IOI, Indirect Object Identification) 회로 재현:**

    - **내용:** "Alice and Bob went to the store. Alice gave a drink to..."라는 문장에서 'Bob'을 예측하는 논리 구조를 찾는 Task입니다.
        
    - **성과:** 과거 연구자들이 수작업으로 찾았던 핵심 노드들의 90% 이상을 자동으로 식별했습니다. 특히, 사람이 간과했던 일부 세부적인 연결(Edge)들까지 찾아내어 회로의 완성도를 높였습니다.
        
- **Greater-Than(크다 비교) 회로 발견:**
    
    - **내용:** "The year was 1990. After a few years, it was 199..."와 같은 문장에서 다음에 올 숫자가 이전 숫자보다 커야 함을 인지하는 논리입니다.
        
    - **성과:** 이 태스크와 관련된 5가지 주요 구성 요소 유형을 100% 찾아냈습니다. 수만 개의 연결 중 단 68개의 핵심 연결만으로 모델의 성능을 거의 완벽하게 유지하는 '최소 회로'를 입증했습니다.
        
- **Python Docstring 회로:**
    
    - **내용:** 파이썬 코드에서 함수의 매개변수 이름을 인식하고 이를 주석(Docstring)에 연결하는 복잡한 논리입니다.
        
    - **성과:** 자연어뿐만 아니라 코드 데이터에서도 모델이 특정 문법적 규칙을 처리하는 하위 네트워크(회로)를 성공적으로 추출할 수 있음을 보여주었습니다.
## 한계
1. 상보적 연결 - 연결을 하나씩 끊어보는데, A, B 연결 모두 있어야 동작하는 기능도 있어서 하나만 끊었을때는 괜찮다고 제거 가능
2. superposition 문제 - 한 노드가 여러 기능을 동시에 수행하면, 상관없는 기능이 딸려 들어오게 되는걸 분리 불가 ex. 덧셈 기능 회로만 분리했다고 생각했는데 그 안에 프랑스어 기능도 같이 들어있음
   $\rightarrow$ [[SAE]]
